Compiler layer

The compiler decides how a task should enter the kernel.

MotifVM now computes consequence-weighted motif risk before execution. A MotifFrame records required support, current support, gap, risk, and selected policies. The ReasoningPlan then chooses passes from declared PassEffect metadata instead of relying on a fixed pass order.

Low-risk tasks select light verification. Authority-heavy audit and security tasks select strict verification, evidence construction, authority extraction, structural checks, and replan behavior when invariants classify a terminal failure.

1. What is MotifVM?

MotifVM is a reasoning kernel, not an agent framework.

MotifVM turns messy inputs and LLM/tool outputs into typed, inspectable state transitions. It does not let a model response become truth by default. Every meaningful change is proposed as a StatePatch, checked against permissions and invariants, then committed as success or structured failure.

The runtime separates perception, reasoning, trust, mutation, and portability: adapters extract evidence and facts, passes propose claims, invariants decide terminal states, and audit packs make the trace portable.

2. Kernel contract

The kernel owns mutation.

No adapter, pass, tool, or LLM call mutates CognitiveState directly. The runtime alone validates, authorizes, applies, verifies, commits, and exports.

The core does not care whether evidence came from a spreadsheet cell, repository line, PDF region, API JSON path, or log event. It cares that the adapter output is stable, hash-bound, and schema-valid.

5. How StatePatches are authorized

Outputs become proposals before they become state.

Role Allowed Blocked
Domain pass claims, outputs, artifacts, domain-scoped graph edges cross-domain evidence injection
LLM bounded narrative artifacts terminal status, invariants, input manifest, authority refs
Reconciliation non-mutating correction proposals source mutation without confirmation
Runtime commit state, write audit metadata unverified terminal claims

6. How invariants define terminal states

Failure is structured runtime output.

MotifVM commits failed states instead of hiding them behind exceptions or vague refusals.

committed_success

All error-severity invariants pass.

input_invalid

Malformed or invalid source artifact values.

computation_blocked

Required evidence is missing or insufficient.

reconciliation_required

Evidence conflicts and a non-mutating correction is proposed.

security_risk_detected

Code-review invariants detect unsafe behavior.

llm_patch_rejected

A model proposed a forbidden transition.

8. How to add a domain

Domains plug into the kernel through profiles, passes, and invariants.

1. Define authority

Add checked-in authority sources and section-level AuthorityRefs.

2. Choose adapters

Use existing adapters or add one that emits EvidenceRefs and ExtractedFacts.

3. Write domain passes

Consume facts and propose claims, outputs, and artifacts through StatePatch.

4. Add invariants

Define terminal correctness: pass, fail, block, reconcile, or reject.

5. Verify packs

Export audit packs and run the verifier in CI before trusting the domain.
